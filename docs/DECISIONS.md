# Журнал решений

## 2026-06-07. Форма проекта

Статус: частично пересмотрено решением "Выбор стека для первой реализации" ниже. Docker-first и итеративный подход остаются актуальными; Kotlin-first больше не актуален.

Решение:

- Развивать проект итеративно, под контролем пользователя.
- Использовать Docker-first подход.
- Первоначально считать Kotlin предпочтительным направлением для основного сервиса, пока лёгкое исследование стека не покажет обратное.
- Считать Codex automation обязательным будущим компонентом.
- Оставить Python как fallback или sidecar для извлечения текста статей и RAG.

Причины:

- Docker-first делает локальную разработку ближе к будущему развёртыванию на VPS.
- Kotlin интересен для изучения и подходит для service orchestration, bot polling, доступа к SQLite и HTTP-вызовов к LLM.
- Python может оказаться сильнее на будущих этапах: article extraction, embeddings и RAG. Поэтому границы модулей должны оставаться заменяемыми.
- Codex automation не должна заменять бота. Она должна периодически ревьюить сохранённые статьи и контекст Obsidian.

Последствия:

- Начинаем с планирования и исследования стека внутри папки проекта.
- Не фиксируем подробную RAG-схему слишком рано.
- Проектируем adapters и services так, чтобы Telegram, VK, automation и будущий RAG могли использовать общую логику обработки статей.

## 2026-06-07. Выбор стека для первой реализации

Решение:

- Выбираем Python как основной и единственный стек первой реализации.
- Выбираем Telegram как единственный канал MVP.
- VK откладываем на будущий adapter.
- Выбираем Docker Compose с начала проекта.
- Выбираем `aiogram` для Telegram polling и отправки ответов.
- Выбираем OpenAI-совместимый Python LLM adapter через переменные окружения.
- Выбираем SQLite через stdlib `sqlite3`, простые SQL-миграции и repository layer.
- Выбираем `trafilatura` для извлечения чистого текста статей.

Причины:

- Python быстрее доведёт MVP до работающего Telegram-бота со статьями, LLM-анализом и SQLite.
- Telegram достаточно для первой полезной версии; VK не должен усложнять стартовый конфиг, архитектуру и критерии готовности.
- `aiogram` закрывает polling и базовую bot-инфраструктуру без ручной реализации Telegram Bot API.
- Python сильнее в извлечении текста статей и будущем RAG, поэтому больше не нужен отдельный Python sidecar.
- stdlib `sqlite3` достаточно прост для начальной схемы; SQLAlchemy можно добавить позже, если запросы и модель усложнятся.

Последствия:

- Этап 2 начинается с Python-каркаса, Dockerfile, `docker-compose.yml`, `.env.example` и минимального запуска приложения.
- Минимальный набор локальных инструментов и Docker-зависимостей фиксируется в `docs/SETUP.md`.
- В MVP нет VK token и VK adapter.
- В коде должны рано появиться интерфейсы для article loading, article extraction, LLM analysis и storage.
- Telegram adapter не должен знать о SQL, LLM-клиенте и деталях извлечения статей.
- Внутренняя модель сообщений остаётся достаточно общей, чтобы VK можно было добавить позже без переписывания article pipeline.
- Подробности исследования зафиксированы в `docs/STACK_RESEARCH.md`.

## 2026-06-27. Миграции до первого MVP

Решение:

- Сохранить реализованный механизм SQL-миграций.
- До выхода первого MVP изменять одну начальную миграцию `0001_initial_schema.sql`.
- После изменения начальной схемы удалять локальную `data/app.db` и создавать базу заново.
- Начать добавлять новые миграции `0002`, `0003` и далее после появления реальных данных, которые нельзя пересоздать.

Причины:

- До реального использования базы начальная схема будет часто меняться.
- Поддержка истории каждой промежуточной версии сейчас не приносит практической пользы.
- Готовый механизм миграций понадобится после появления ценных пользовательских данных.

Последствия:

- До первого MVP локальная база считается временной и может быть удалена.
- Изменения `0001_initial_schema.sql` требуют пересоздания локальной базы.
- После появления реальных данных применённые миграции больше не изменяются.
